# 🎓 School Fee Management System

---

## 🧠 1. Introduction

The **School Fee Management System** is a software application designed to automate fee collection, tracking, and allocation for students.

It replaces manual fee handling with a **digital, transparent, and scalable system** that:

* Tracks student fees
* Handles flexible payments
* Automatically allocates payments
* Maintains complete financial records

---

## 🎯 2. Objective

The system is designed to:

* Allow **flexible payments** (any amount)
* Automatically allocate payments across fee components
* Track:

  * Total fee
  * Paid amount
  * Remaining balance
* Maintain a **complete audit trail**
* Support **class-wise dynamic fee configuration**

---

## ⚙️ 3. Key Features

### ✅ Core Features

* Class-based fee structure
* Multiple fee components (Tuition, Transport, etc.)
* Priority-based payment allocation
* Partial payment support
* Payment history tracking

### ✅ Advanced Features

* Real-time balance calculation
* Automatic payment splitting
* Class-specific fee customization
* Scalable architecture for future expansion

## 🗄️ 5. Database Design

### 📌 Tables

1. **students**
   Stores student details

2. **classes**
   Stores class information

3. **fee_components**
   Defines global fee types (Tuition, Transport, etc.)

4. **class_fee_structure ⭐**
   Defines:

   * Fee amount per class
   * Applicable components
   * Priority order

5. **payments**
   Stores each payment made

6. **payment_allocations ⭐**
   Stores how each payment is distributed across components

---

### 🧠 Key Design Concepts

#### ✅ Separation of Concerns

* `fee_components` → global definitions
* `class_fee_structure` → class-specific configuration

#### ✅ Allocation Tracking

* Every payment is broken down and stored in `payment_allocations`

---

## 🔄 6. Core Business Logic

### 🎯 Payment Allocation Logic

When a student makes a payment:

1. Fetch class fee structure (sorted by priority)
2. Calculate remaining due for each component
3. Allocate payment accordingly

---

### 🧾 Algorithm

```
remaining = payment_amount

FOR each component (priority order):
    due = total - paid

    IF remaining >= due:
        allocate = due
    ELSE:
        allocate = remaining

    save allocation
    remaining -= allocate

    IF remaining == 0:
        BREAK
```

---

### 💡 Example

| Component | Total | Allocated |
| --------- | ----- | --------- |
| Tuition   | 2000  | 2000      |
| Transport | 1000  | 1000      |
| Library   | 500   | 500       |

**Payment:** ₹3500 → Fully distributed

## 📡 8. API Endpoints

---

### 💳 Payment APIs

#### ➤ Create Payment

```
POST /payments/pay
```

#### ➤ Get Student Details

```
GET /students/{id}
```

---

### 🧱 Fee Structure APIs (NEW)

#### ➤ Create Class Fee Structure

```
POST /class-fee-structure
```

**Request Body:**

```json
{
  "class_id": 1,
  "fees": [
    {
      "component_id": 1,
      "amount": 2000,
      "priority": 1
    },
    {
      "component_id": 2,
      "amount": 1000,
      "priority": 2
    }
  ]
}
```

---

#### ➤ Update Class Fee Structure

```
PUT /class-fee-structure/{class_id}
```

👉 Recommended approach:

* Delete existing records
* Insert updated structure

---

#### ➤ Get Class Fee Structure

```
GET /class-fee-structure/{class_id}
```
